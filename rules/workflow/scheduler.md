# スケジューラー運用ガイド

## 基本思想

**Desktop で育てて、Cloud に昇格させる。**
いきなり Cloud に登録すると、品質の低いスキルが垂れ流されるリスクがある。
段階的に安定性を確認してから自動化を本番化する。

---

## 3種類のスケジューラーの使い分け

| 種類 | 実行環境 | 適した用途 |
| --- | --- | --- |
| `/loop` | ローカル（セッション内） | デプロイ監視・ビルド完了待機など、セッション内の一時的なポーリング |
| Desktop | ローカルマシン | 微調整中のタスク・PDCAサイクル中の作業 |
| Cloud | Anthropic クラウド | 安定稼働済みの完全自動タスク（最小実行間隔: 1時間） |

---

## Desktop → Cloud 昇格フロー

```
Phase 1: スキル作成
  手動で /skill-name を実行し、出力内容を確認する
        ↓
Phase 2: Desktop スケジューラーで試運転
  実行のたびに結果を確認し、以下を調整する
  ・プロンプトの微調整（出力フォーマット・情報の粒度）
  ・エッジケース対応（データ取得失敗時のフォールバック）
  ・実行タイミングの最適化
        ↓
Phase 3: Cloud スケジューラーに昇格（下記チェックリストを満たした場合のみ）
```

---

## Cloud 昇格チェックリスト

以下を**すべて満たした**タスクのみ Cloud に昇格させる。

- [ ] 手動介入なしで **3回以上連続**で期待通りの出力が得られた
- [ ] 出力フォーマットが安定している
- [ ] データ取得失敗・空データ時の**フォールバック処理**が定義されている
- [ ] 実行結果の通知先（Slack DM・チャンネル等）が設定されている

---

## 自動化に向いているタスクの選定基準

**自動化すべき**:
- 手順が明確で、毎回同じ操作をする
- 繰り返し頻度が高い（週1回以上）
- 判断が不要、または判断基準が明文化できる

**自動化すべきでない**:
- コンテキストに強く依存する判断が必要
- 出力品質のばらつきが許容できない
- 結果の事後確認では間に合わない（リアルタイムの判断が必要）

---

## スケジューラーが強力な理由

Claude Code のスケジューラーは、接続済みの MCP をそのまま利用できる。
通常の cron ジョブと異なり、Slack・Notion・GitHub・Google Calendar などの
複数サービスを横断するワークフローを、自然言語のプロンプト1つで定期実行できる。

```
例: 毎朝 8:30 に実行
  1. Google Calendar から当日の予定を取得
  2. 前日の未完了 TODO を Notion から抽出
  3. 日次レポートを生成して Slack に投稿
```

個別の API クライアントを書く必要がなく、認証管理も不要。
