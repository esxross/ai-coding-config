# コードレビュー品質ルール

## レビューの目的

- バグ・セキュリティ問題の早期発見
- プロジェクト固有ルールへの準拠確認
- 設計の一貫性維持

## レビュー対象の優先順位

1. バグ・セキュリティ脆弱性（必ず指摘）
2. パフォーマンス問題（ループ内DB/APIアクセス等）
3. 型の整合性・コンパイルエラー
4. コーディングルール違反
5. 設計・命名の改善提案

## コードレビューで必ず確認すること

### 差分だけでなく全体を見る

- 変更ファイルの全体（既存コードとの整合性）
- 変更したメソッドの利用箇所（影響範囲の確認）
- 関連する型定義・インターフェース

### セキュリティ

- ユーザー入力のバリデーション漏れ
- APIキー・シークレットのハードコード
- 認証・認可チェックの漏れ
- SQLインジェクション・XSS等のインジェクション

### パフォーマンス

- ループ内でのDB/外部APIアクセス
- トランザクション内での外部API呼び出し
- N+1クエリ

### 型安全性

- TypeScriptコンパイルエラーになるimport
- `any` の不適切な使用
- 型アサション（`as`）の乱用

## プロジェクト固有のレビュー観点の追記方法

各プロジェクトの `CLAUDE.md` または `instructions/*.md` に以下の形式で追記してください：

```markdown
## コードレビューで必ず指摘する内容

- [具体的な観点]: [どのような状態の場合に] [何を指摘・提案するか]
  - 例: クラス名が抽象的な場合（例: UserUseCase）、クラス内部の実装を確認して具体的なクラス名を提案する
```

## 指摘フォーマット（標準）

```
### [{重大度}] {ファイルパス}#{行番号}

- **ルール**: {ルールIDまたは観点名}
- **内容**: {問題の説明と影響}
- **修正提案**: {具体的な修正方法}
```

## 重大度の定義

| レベル | 基準 |
|--------|------|
| HIGH | バグ・セキュリティ・データ整合性・コンパイルエラー・本番障害リスク |
| MEDIUM | パフォーマンス問題・設計上の問題・保守性に大きく影響する問題 |
| LOW | コーディング規約違反・命名の改善提案・軽微なリファクタリング |

## 避けるべき指摘パターン

- 根拠なしのスタイル好み（「〜の方が良い気がします」）
- 差分外の既存コードへの指摘
- 修正提案のない HIGH 指摘
- 推測・仮定に基づく指摘（「〜かもしれません」）
- LOW 指摘の過多（5件以上は箇条書きに集約する）
