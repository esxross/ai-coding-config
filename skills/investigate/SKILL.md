# /investigate スキル

## 概要

**「調査なし修正禁止」原則**を強制するデバッグスキルです。
バグを発見してもすぐにコードを変更しません。
根本原因を特定してから修正することで、「直したら別のバグが出た」を防ぎます。

---

## 実行手順

### Step 1: 症状の記録

まず問題を正確に言語化する。

```
- 何が起きているか（実際の動作）
- 何が期待されるか（期待する動作）
- いつから発生しているか
- 再現条件（常に？特定の入力で？）
```

修正の仮説を立てるのはこのステップ完了後。

---

### Step 2: 変更履歴の確認

問題が発生しているファイル・モジュールの変更履歴を追う。

```bash
# 対象ファイルの最近の変更
git log --oneline -20 -- {対象ファイル}

# 特定コミットの差分
git show {commit-hash}

# いつ導入されたか二分探索
git bisect start
git bisect bad HEAD
git bisect good {正常だったcommit}
```

「いつから壊れたか」を特定することが根本原因の最短経路。

---

### Step 3: 影響範囲の把握

問題のコードがどこで使われているかを確認する。

```bash
# 関数・変数の使用箇所
grep -r "{関数名}" src/ --include="*.ts"

# インターフェースの実装箇所
grep -r "implements {インターフェース名}" src/

# テストが存在するか
find . -name "*.test.*" | xargs grep "{対象}"
```

修正が影響する箇所をすべて把握してから変更する。

---

### Step 4: 仮説の立案と検証

根本原因の仮説を最大3つ立て、証拠で検証する。

```
仮説1: {原因の候補}
  → 証拠: {git log / ログ / テスト結果}
  → 検証: {確認コマンドまたは手順}

仮説2: {原因の候補}
  → 証拠: ...
  → 検証: ...
```

証拠なしに「おそらくここが原因」と言わない。

---

### Step 5: 修正方針の確定

根本原因が特定できたら修正方針を宣言してから実装する。

```
根本原因: {特定された原因}
修正方針: {何をどう変えるか}
影響ファイル: {変更するファイル一覧}
リグレッションリスク: {壊れる可能性がある箇所}
```

方針を確定する前にコードを変更しない。

---

### Step 6: 修正と検証

方針に従い最小限の変更を実施し、テストで動作を確認する。

```bash
# 既存テストの実行
npm test -- {対象ファイル}

# 修正前後の動作比較
git diff
```

修正後に Step 3 の影響範囲で副作用がないことを確認する。

---

## アンチパターン（やってはいけないこと）

- 原因を特定する前にコードを変更する
- 「とりあえず直してみる」を繰り返す
- テストを書かずに修正完了とする
- 影響範囲を確認せずにリリースする
