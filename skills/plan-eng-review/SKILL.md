# /plan-eng-review スキル

## 概要

実装前の設計をシニアエンジニア視点で**批判的に**レビューします。
`/architect-first` で作成した設計案、または任意の設計ドキュメントを受け取り、
「この設計は本当に正しいか？」を実装ロック前に検証します。

**READ-ONLY**: 設計への指摘のみ行い、コードは変更しません。

---

## 入力

以下のいずれかを渡す：

1. `/architect-first` の出力（設計 Markdown）
2. 設計ドキュメントのファイルパス
3. チャットに直接貼り付けた設計テキスト

---

## レビュー観点

### 1. アーキテクチャの健全性

- **単一責任**: 各コンポーネントの責務は明確か？複数の関心事が混在していないか
- **依存関係の方向**: 循環依存・不適切な層間参照がないか
- **インターフェース設計**: 公開 API は最小限か。変更が伝播しにくいか
- **状態管理**: 状態はどこに置かれるか。整合性は保たれるか

### 2. テスト戦略

- ユニットテスト・統合テスト・E2E テストの境界が明確か
- テストが難しい設計（密結合・副作用・グローバル状態）になっていないか
- モック・スタブが多用される設計になっていないか（本番との乖離リスク）

### 3. スケーラビリティ・パフォーマンス

- ボトルネックになりうる箇所（N+1、同期処理、単一障害点）はあるか
- データ量・リクエスト数が10倍になった時に破綻しないか
- キャッシュ・非同期化が必要な箇所を見落としていないか

### 4. 技術的負債リスク

- 「今は動くが後で困る」設計はないか
- 変更しやすい設計か（インターフェースが安定しているか）
- 過剰な抽象化・早すぎる汎用化になっていないか（YAGNI）
- 省略しているエラーハンドリングはないか

### 5. セキュリティ・観点の漏れ

- 認証・認可の考慮があるか
- 外部入力のバリデーションポイントが設計に含まれているか
- 機密データの扱い方が設計に明記されているか

---

## 出力形式

```markdown
## Engineering Review: {設計タイトル}

### 総評
{設計全体の評価を2〜3文で}

---

### 指摘事項

#### [BLOCKING] {問題タイトル}
- **観点**: {アーキテクチャ / テスト / スケーラビリティ / 負債 / セキュリティ}
- **問題**: {何が問題で、なぜ実装前に解決すべきか}
- **代替案**: {推奨する設計パターンまたは修正方針}

#### [WARNING] {問題タイトル}
- **観点**: ...
- **問題**: ...
- **提案**: {やった方がよい改善案}

#### [NOTE] {気づき}
- 問題ではないが設計判断として明示しておくべき点

---

### 実装ロック判定

✅ **実装可能** — 重大な問題なし
⚠ **要修正後に実装** — BLOCKING 指摘を解消してから進む
🚫 **設計見直しが必要** — アーキテクチャレベルで再検討が必要
```

---

## architect-first との連携

推奨フロー：

```
/architect-first  →  設計案を提示（実装前の設計）
       ↓
/plan-eng-review  →  設計を批判的にレビュー（BLOCKING があれば設計修正）
       ↓
実装開始（設計ロック）
```
