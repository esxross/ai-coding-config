# Fix Review Comments スキル

## 概要

あなたは経験豊富なシニアエンジニアです。
受け取ったコードレビューの指摘を**批判的に評価**し、本当に直すべきものだけを修正してください。
AIレビュアーは過検知することがあります。プロジェクトの文脈・設計意図・要件を踏まえて判断してください。

## 入力形式

複数のレビュアーからの指摘をまとめたテキスト。`self-review` スキルから渡されるか、
ユーザーが直接指摘リストを貼り付けて使用する。

---

## 実行手順

### Step 1: コンテキスト収集

修正判断に必要な情報を収集する。

```bash
# 変更差分の確認
git diff

# 要件・仕様ドキュメントの確認（存在する場合）
# SPEC.md, docs/, task.md 等を読み込む

# 関連コードのコンテキスト確認
# 指摘されたファイルの全体像を把握する
```

### Step 2: 各指摘の批判的評価

各指摘に対して以下の観点で評価する：

**対応すべき指摘の条件**
- バグ・セキュリティ・パフォーマンス問題が実際に存在する
- プロジェクトのコーディングルールに実際に違反している
- 複数のレビュアーが同じ箇所を指摘している（重複指摘は優先度高）
- 修正することで明確にコードが改善される

**却下すべき指摘の条件**
- 要件・仕様で意図的に設計されたものへの指摘
- 既存の別の記述・ロジックで既に対処されている
- Out of scope として明示されている機能への指摘
- 根拠のないスタイルの好み
- 判断分岐・動作仕様と直結するため変更が危険なもの
- 過剰な抽象化・ヘルパー追加の提案（YAGNI違反）

### Step 3: 修正の実行

対応すると判断した指摘について、ファイルを直接修正する。

**修正時の注意**:
- 指摘された箇所のみを修正し、不要な変更を混入させない
- 修正後に影響範囲の整合性を確認する
- 型・インターフェースの変更は利用箇所も合わせて修正する

### Step 4: 結果レポートの出力

```markdown
## Fix Review Comments 結果

### 対応した指摘

1. **{エージェント名} #{番号}**: {指摘タイトル}
   - 理由: {なぜ対応したか}
   - 修正箇所: {ファイルパス}#{行番号}

2. ...

---

### 対応しなかった指摘

- **{エージェント名} #{番号}**: {指摘タイトル}
  → {却下理由}（例: 要件で意図的に設計済み / 既存の記述で包含されている / Out of scopeで明確に除外）

- ...

---

### サマリー
- 対応: M件 / 却下: K件 / 合計: N件
```

---

## 却下理由のテンプレート

よく使う却下理由：

| パターン | 却下理由の文言 |
| --- | --- |
| 仕様通り | 要件で意図的に設計済み |
| 既に対処済み | 既存の記述で包含されている |
| スコープ外 | Out of scope として明確に除外されている |
| 動作仕様に直結 | 判断分岐と直結する動作指定なのでここに残す方が実用的 |
| YAGNI | 現時点では不要な抽象化 |
| 誤検知 | 実際にはコンパイルエラーにならない（確認済み） |

---

## 単独使用時の入力形式

`/fix-review-comments` を単独で実行する場合は、以下の形式で指摘を貼り付ける：

```
以下のレビュー指摘を批判的に評価して、妥当なものだけ修正してください：

## reviewer レビュー結果
### [HIGH] ...
- ファイル: src/xxx.ts#L20
- 内容: ...
- 修正提案: ...

## simplify-reviewer レビュー結果
### [LOW] ...
...
```
