# /cso スキル（Chief Security Officer レビュー）

## 概要

OWASP Top 10 と STRIDE 脅威モデルに基づき、コード変更をセキュリティ観点でレビューします。
READ-ONLY スキルです。指摘のみ行い、ファイルの修正は行いません。

**推奨使用タイミング**:
- 認証・認可コードの変更時
- 決済・個人情報を扱う機能の変更時
- API エンドポイントの追加・変更時
- 本番リリース前の最終確認

---

## 実行手順

### Step 1: 変更差分の取得

```bash
git diff HEAD          # 未コミットの変更
git diff HEAD~1 HEAD   # 直近のコミット
```

### Step 2: OWASP Top 10 チェック

各項目を変更差分に対して確認する。

| # | カテゴリ | チェック内容 |
| --- | --- | --- |
| A01 | アクセス制御の不備 | 認可チェックの漏れ・水平/垂直権限昇格 |
| A02 | 暗号化の失敗 | 機密データの平文送信・弱いハッシュアルゴリズム |
| A03 | インジェクション | SQL・コマンド・LDAP・XPath インジェクション |
| A04 | 安全でない設計 | 脅威モデリングの欠如・セキュリティ設計の不備 |
| A05 | セキュリティ設定ミス | デフォルト認証情報・不要な機能の有効化 |
| A06 | 脆弱なコンポーネント | 既知の脆弱性を持つライブラリの使用 |
| A07 | 認証の失敗 | セッション管理の不備・ブルートフォース対策の欠如 |
| A08 | ソフトウェアの完全性 | 署名なし更新・安全でない CI/CD パイプライン |
| A09 | ロギングの失敗 | セキュリティイベントの記録漏れ・ログへの機密情報混入 |
| A10 | SSRF | サーバーサイドリクエストフォージェリ |

### Step 3: STRIDE 脅威モデリング

変更されたコンポーネントに対して STRIDE 観点で確認する。

| 脅威 | 概要 | チェック内容 |
| --- | --- | --- |
| **S**poofing | なりすまし | 認証が正しく機能しているか |
| **T**ampering | 改ざん | 入力値の検証・整合性チェックがあるか |
| **R**epudiation | 否認 | 監査ログが十分に記録されているか |
| **I**nformation Disclosure | 情報漏洩 | エラーメッセージ・ログに機密情報が含まれないか |
| **D**enial of Service | サービス拒否 | レート制限・リソース消費の上限があるか |
| **E**levation of Privilege | 権限昇格 | 意図しない権限取得が可能でないか |

### Step 4: プロジェクト固有チェック

`CLAUDE.md` に記載のセキュリティ要件があれば追加で確認する。

```markdown
## /cso 追加チェック（CLAUDE.md への記載例）
- RLS ポリシーの変更はすべてレビュー対象
- 外部 API キーを使う処理は環境変数経由であること
```

## 出力形式

```markdown
## CSO セキュリティレビュー {YYYY/MM/DD}

### [CRITICAL] {脆弱性タイトル}
- **分類**: OWASP A{番号} / STRIDE {脅威}
- **ファイル**: {パス}#{行番号}
- **内容**: {脆弱性の説明と攻撃シナリオ}
- **修正方針**: {具体的な対処法}

### [HIGH] ...

### [MEDIUM] ...

---

### サマリー
- CRITICAL: X件
- HIGH: Y件
- MEDIUM: Z件
- リリース判定: ✅ 問題なし / ⚠ 要修正 / 🚫 リリース不可
```

問題なしの場合:

```markdown
## CSO セキュリティレビュー {YYYY/MM/DD}
セキュリティ上の問題は検出されませんでした。
リリース判定: ✅ 問題なし
```
