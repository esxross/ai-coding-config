# Fix Review Comments スキル

## 概要

受け取ったコードレビューの指摘を**批判的に評価**し、本当に直すべきものだけを修正します。
AIレビュアーは過検知することがあります。プロジェクトの文脈・設計意図・要件を踏まえて判断してください。

---

## 実行手順

### Step 1: コンテキスト収集

```bash
git diff
# SPEC.md, docs/, task.md 等を読み込む
# 指摘されたファイルの全体像を把握する
```

### Step 2: 各指摘の批判的評価

**対応すべき指摘の条件**
- バグ・セキュリティ・パフォーマンス問題が実際に存在する
- プロジェクトのコーディングルールに実際に違反している
- 複数のレビュアーが同じ箇所を指摘している（重複指摘は優先度高）
- 修正することで明確にコードが改善される

**却下すべき指摘の条件**
- 要件・仕様で意図的に設計されたものへの指摘
- 既存の別の記述・ロジックで既に対処されている
- Out of scope として明示されている機能への指摘
- 根拠のないスタイルの好み
- 判断分岐・動作仕様と直結するため変更が危険なもの
- 過剰な抽象化・ヘルパー追加の提案（YAGNI違反）

### Step 3: 修正の実行

対応すると判断した指摘についてファイルを修正する。
指摘された箇所のみを修正し、不要な変更を混入させない。

### Step 4: 結果レポートの出力

```markdown
## Fix Review Comments 結果

### 対応した指摘

1. **{エージェント名} #{番号}**: {指摘タイトル}
   - 理由: {なぜ対応したか}
   - 修正箇所: {ファイルパス}#{行番号}

### 対応しなかった指摘

- **{エージェント名} #{番号}**: {指摘タイトル}
  → {却下理由}

### サマリー
- 対応: M件 / 却下: K件 / 合計: N件
```

---

## 却下理由テンプレート

| パターン | 文言 |
| --- | --- |
| 仕様通り | 要件で意図的に設計済み |
| 既に対処済み | 既存の記述で包含されている |
| スコープ外 | Out of scope として明確に除外されている |
| 動作仕様に直結 | 判断分岐と直結する動作指定なのでここに残す方が実用的 |
| YAGNI | 現時点では不要な抽象化 |
| 誤検知 | 実際にはコンパイルエラーにならない（確認済み） |
